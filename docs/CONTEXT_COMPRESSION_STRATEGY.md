# 三级上下文压缩策略

> 通用方案文档，适用于任何需要长对话上下文管理的 Agent 系统。

## 问题背景

Agent 长对话中，消息历史持续增长，最终超过 LLM 上下文窗口限制。单纯的"截断最近 N 条"会丢失早期重要信息，而"不做处理"会导致 token 溢出。本方案提出一种渐进式的三级压缩架构，在信息保留与 token 效率之间取得平衡。

---

## 核心思想

通过**渐进式压缩**，在不同 token 压力级别下采用不同策略，逐步牺牲细节、保留核心：

```
Level 1: 回合摘要（轻度压缩）
  ↓ 摘要数量累积到阈值
Level 2: 阶段概览（中度压缩）
  ↓ 阶段概览仍然超限
Level 3: 滑动窗口 + 全局概览（重度压缩）
```

---

## Level 1: 回合摘要

### 目标

压缩每轮对话中的执行细节，保留用户意图和执行结果。

### 策略

| 维度 | 说明 |
|------|------|
| **粒度** | 每轮用户消息 → Agent 执行过程 |
| **压缩方式** | 将用户消息之后的所有 assistant/tool 消息压缩为一段执行摘要 |
| **保留内容** | 所有用户消息完整保留 |
| **信息损失** | 低（仅丢失工具调用的中间细节） |

### 触发条件

当消息历史的 token 数量超过设定阈值时触发。可采用双重检查：
- 本地 token 估算（实时计算）
- API 返回的精确 token 数（LLM 调用后更新）

### 压缩后结构

```
原始：
system → user1 → [assistant/tool × N] → user2 → [assistant/tool × N] → user3

压缩后：
system → user1 → summary1 → user2 → summary2 → user3
```

### 实现要点

1. 按用户消息分割对话回合
2. 对每回合的执行过程调用 LLM 生成摘要（独立的新对话，不依赖当前上下文）
3. 摘要以特殊格式插入（如 `[Execution Summary] {摘要内容}`）
4. 摘要完成后需设置冷却期，避免连续触发

### 失败兜底

若 LLM 摘要生成失败，退回纯文本摘要（简单拼接工具调用名称和结果预览）。

---

## Level 2: 阶段概览

### 目标

当回合摘要数量累积过多时，将早期多轮摘要进一步压缩。

### 策略

| 维度 | 说明 |
|------|------|
| **粒度** | 多轮回合（如 5-10 轮） |
| **压缩方式** | 将最早的 N 个回合摘要压缩为一个阶段概览 |
| **保留内容** | 阶段概览 + 最近 N 轮完整摘要 |
| **信息损失** | 中（早期回合的具体结果被概括） |

### 触发条件

当回合摘要数量超过设定阈值（如 5 个）时触发。

### 压缩后结构

```
压缩前：
[user1][摘要1][user2][摘要2][user3][摘要3][user4][摘要4][user5][摘要5][user6][摘要6]

压缩后：
[阶段概览: "前5轮完成了模块A的重构和测试"][user6][摘要6]
```

### 实现要点

1. 识别最早的 N 个回合摘要
2. 调用 LLM 生成阶段概览，提示词聚焦"核心成果"而非细节
3. 用阶段概览替换早期回合（用户消息 + 摘要）
4. 保留最近几轮完整摘要，确保近期上下文质量

---

## Level 3: 滑动窗口 + 全局概览

### 目标

当阶段概览 + 近期摘要仍然超限时，进入最终压缩阶段。

### 策略

| 维度 | 说明 |
|------|------|
| **粒度** | 整个会话 |
| **压缩方式** | 生成全局概览，只保留最近 N 轮对话 |
| **保留内容** | 全局概览 + 最近上下文 |
| **信息损失** | 高但可控（早期细节完全丢失，但核心进展保留） |

### 触发条件

当阶段概览 + 近期摘要的 token 数量仍然超过阈值时触发。

### 压缩后结构

```
压缩前：
[阶段概览: "完成了模块A重构"][user6][摘要6]...[user15][摘要15]

压缩后：
[全局概览: "用户完成了auth模块重构，正在编写测试用例"][user13][摘要13][user14][摘要14][user15][摘要15]
```

### 实现要点

1. 调用 LLM 生成全局概览，涵盖整个会话的核心进展和当前状态
2. 滑动窗口：只保留最近 N 轮对话（用户消息 + 执行摘要）
3. 最终结构：`[system] + [全局概览] + [最近 N 轮]`

---

## 数据流示意

```
原始对话
  ↓ Level 1 (每轮执行过程 → 摘要)
[user1][摘要1][user2][摘要2][user3][摘要3]...[user10][摘要10]
  ↓ Level 2 (摘要数量 > 5)
[阶段概览: "前5轮完成了重构和测试"][user6][摘要6]...[user10][摘要10]
  ↓ Level 3 (仍然超限)
[全局概览: "用户完成了auth模块重构"][user8][摘要8][user9][摘要9][user10][摘要10]
```

---

## 对比分析

| 维度 | 传统滑动窗口 | 仅 Level 1 | 完整三级方案 |
|------|-------------|-----------|-------------|
| 保留内容 | 最近 N 条消息 | 所有用户意图 + 执行摘要 | 全局概览 + 近期上下文 |
| 信息丢失 | 早期对话完全丢弃 | 仅丢失执行细节 | 早期细节丢失，概览保留 |
| 连贯性 | 可能丢失上下文 | 用户意图完整保留 | 可控的信息损失 |
| Token 效率 | 线性增长 | 摘要替代详细过程 | 多层压缩，效率最高 |
| 额外成本 | 无 | 每次触发额外调用一次 LLM | Level 2/3 额外调用 |

## 配置参数建议

```yaml
context_management:
  token_limit: 80000              # Level 1 触发阈值
  summary_threshold: 5            # Level 2 触发阈值（摘要数量）
  max_recent_rounds: 10           # Level 3 保留最近轮数
  enable_phase_summary: true      # 是否启用阶段概览
  enable_sliding_window: true     # 是否启用滑动窗口
  summary_max_words: 1000         # 单条摘要最大字数
  cooling_period: true            # 是否启用摘要冷却期
```

---

## 实施建议

### 优先级

1. **Level 1（回合摘要）**：必做。实现成本适中，能解决大部分长对话场景。
2. **Level 2（阶段概览）**：中等优先级。当对话超过 5-10 轮时触发。
3. **Level 3（滑动窗口）**：低优先级。仅在极端长对话场景下需要。

### 注意事项

1. **摘要质量**：Level 1 的摘要质量直接影响后续层级，提示词需精心设计
2. **冷却机制**：每次摘要完成后需跳过下一次 token 检查，避免连续触发
3. **角色设计**：摘要消息建议使用特殊角色或格式标记，便于后续层级识别
4. **失败处理**：LLM 摘要生成可能失败，需有纯文本兜底方案
